1 — Fundamentos do Git

O que é: sistema de controle de versão distribuído; conceitos: snapshot, commit, branch, merge.

Conceitos chave ▼
  • Repositório: local (.git) ou remoto (GitHub).
  • Commit: snapshot do projeto com mensagem.
  • Branch: linha de desenvolvimento independente.
  • Merge vs Rebase: integrar mudanças; rebase reescreve histórico.
Instalação e configuração ▼

Comandos iniciais:

git --version
git config --global user.name "Seu Nome"
git config --global user.email "seu@exemplo.com"

2 — Fluxo de Trabalho Local

Operações diárias para trabalhar em um projeto localmente.

Inicializar e snapshot ▼
git init # cria repositório local
git add # adiciona ao staging
git commit -m "mensagem" # cria commit
git status # ver estado do repositório
git log --oneline --graph # histórico resumido
Branches e trabalho isolado ▼
git branch # lista branches
git checkout -b feature/x # cria e troca para branch
git merge feature/x # mescla branch na atual
git rebase main # reaplica commits sobre main

Dica: faça commits pequenos e frequentes; mensagens claras.

Ignorar arquivos e hooks ▼

Use .gitignore para evitar commitar arquivos binários, chaves e dependências.

# Exemplo .gitignore
node_modules/
.DS_Store
*.log
.env

3 — Integração com GitHub

Subir código para remoto, clonar, sincronizar e criar releases.

Criar e conectar repositório ▼
git remote add origin git@github.com:usuario/repo.git
git push -u origin main # primeiro push estabelece upstream

Também é possível usar HTTPS: https://github.com/usuario/repo.git

Clonar, pull e fetch ▼
git clone git@github.com:usuario/repo.git
git pull origin main # puxa e faz merge automático
git fetch origin # busca referências sem mesclar
git merge origin/main # mescla manualmente
Releases e tags ▼
git tag -a v1.0 -m "Versão 1.0"
git push origin v1.0

Use releases do GitHub para distribuir binários e notas de versão.

4 — Colaboração e Branches

Como trabalhar em equipe: forks, pull requests, revisão de código e resolução de conflitos.

Fluxo simples de colaboração ▼
  1. Criar branch por tarefa: feature/login
  2. Fazer commits pequenos e push para remote
  3. Abrir Pull Request (PR) no GitHub
  4. Revisão por colegas e merge após aprovação
Resolvendo conflitos ▼

Ao haver conflito no merge, o Git marcará os arquivos. Edite, teste e depois:

git add arquivo_resolvido
git commit -m "resolvido conflito"

Ferramentas como VS Code e GitHub Desktop mostram conflitos visualmente.

Pull Request — boas práticas ▼
  • Descrever objetivo e mudanças no PR
  • Referenciar issues relacionadas
  • Manter PRs pequenos e focados
  • Adicionar testes e screenshots quando aplicável

5 — Git & GitHub no VS Code

Extensões úteis: GitLens, Git Graph, GitHub Pull Requests and Issues.

Setup básico ▼
  1. Abra a pasta do projeto no VS Code
  2. Use o painel de fonte de controle para stage/commit
  3. Instale GitLens para histórico e blame

Com a extensão GitHub, é possível abrir/gerenciar PRs diretamente no editor.

Fluxo com GitHub Codespaces / GitHub CLI ▼

Opcionalmente, use gh para criar PRs e issues via terminal:

gh auth login
gh repo clone usuario/repo
gh pr create --fill

6 — Boas Práticas

Mensagens de commit ▼

Use mensagens claras e padronizadas. Exemplo:

feat(auth): adicionar login com Google
fix(ui): corrigir overflow na tela inicial
chore: atualizar dependências
Branching model simples ▼
  • main — código pronto para produção
  • develop — integrações e testes
  • feature/* — novas funcionalidades

Adote convenções como Conventional Commits e semântica de versionamento (SemVer).

7 — Projetos Práticos (exercícios)

  1. Inicie um repositório e publique no GitHub (incluir README e .gitignore).
  2. Crie uma branch feature/exercicio1, implemente e abra PR.
  3. Simule conflito: dois branches editam o mesmo arquivo e resolva o conflito.
  4. Crie uma tag e publique uma release com notas.
